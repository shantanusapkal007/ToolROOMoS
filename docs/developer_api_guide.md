# Developer & API Guide

ToolRoom ERP is built with an API-first methodology. Every action available in the frontend UI is accessible via our RESTful APIs, allowing for deep integration with third-party software, IoT devices, and external analytics platforms.

## 1. Authentication & Security

### 1.1 Acquiring Tokens
All API requests require a JSON Web Token (JWT) passed in the `Authorization` header.
- **Endpoint:** `POST /api/v1/auth/login`
- **Payload:** `{"email": "user@company.com", "password": "securepassword"}`
- **Response:** Returns an `access_token` (valid for 1 hour) and a `refresh_token`.

```bash
curl -H "Authorization: Bearer <your_access_token>" https://api.toolroom.com/v1/work-orders
```

### 1.2 Rate Limits
To protect the system, API requests are throttled.
- **Standard Tier:** 100 requests per minute per IP address.
- Headers included in responses: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`.

## 2. Safe Writes (Idempotency)

For all state-mutating requests (`POST`, `PUT`, `PATCH`, `DELETE`), we highly recommend passing an `X-Idempotency-Key` header.
- The key should be a unique UUID v4 generated by your client.
- If a network failure occurs and you retry the request, the server will recognize the key and safely return the cached successful response rather than duplicating the transaction (e.g., preventing double-deducting inventory).

## 3. Webhooks & Event Streams

Instead of polling our APIs for changes, you can subscribe to real-time Webhooks.

### 3.1 Subscribing to Events
Navigate to `Developer -> Webhooks` in the UI to register a destination URL. You can subscribe to specific domain events:
- `inventory.stock.adjusted`
- `production.machine.offline`
- `sales.quotation.approved`

### 3.2 Payload Structure
When an event occurs, ToolRoom ERP sends an HTTP `POST` to your URL with a signed payload:
```json
{
  "event_id": "evt_987654321",
  "event_type": "production.machine.offline",
  "created_at": "2026-06-29T10:00:00Z",
  "data": {
    "machine_id": "VMC-01",
    "reason_code": "SPINDLE_FAULT"
  }
}
```
*Note: Verify the `X-Signature` header using your webhook secret to ensure the payload originated from ToolRoom ERP.*

## 4. API Error Codes

We use standard HTTP status codes combined with detailed JSON error messages:
- **400 Bad Request:** Missing required fields or invalid data types.
- **401 Unauthorized:** Missing or expired JWT token.
- **403 Forbidden:** The authenticated user lacks the required RBAC permissions to perform the action.
- **409 Conflict:** Idempotency key mismatch or concurrent version modification error (Stale Data).
- **422 Unprocessable Entity:** Business logic validation failure (e.g., trying to release a Work Order without sufficient inventory).

## 5. Database Architecture

### 5.1 Data Dictionary & Module Dependencies
The monolith database is logically partitioned by module. Direct SQL access is prohibited in production; all data extraction should occur via the API.
- **`core` schema:** Contains Tenant, Company, User, and Role tables.
- **`inventory` schema:** Contains Item Master, Stock Ledger, GRN, and Warehouses. Depends on `core`.
- **`production` schema:** Contains Work Orders, Routings, and Machine Telemetry. Depends on `inventory`.

### 5.2 Entity Relationships (ER)
*   A `Company` has many `Plants`.
*   A `Plant` has many `Warehouses` and `Work Centers`.
*   A `Work Order` belongs to a `Plant` and consumes `Items` from a `Warehouse`.
